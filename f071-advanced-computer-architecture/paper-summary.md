# 고급컴퓨터구조 발표 논문 정리
## 용어 정리
- FTL(Flash Translation Layer) : SSD 컨트롤러 내부적으로, HDD와의 호환성을 위한 논리적인 섹터 주소를 실제 SSD의 페이지, 블록으로 매칭시켜주는 레이어.
- LPN(Logical Page Number) : 논리적인 페이지 주소
- LBN(Logical Block Number) : 논리적인 블럭 주소
- DGN(Data Group Number) : 논리적인 LBN의 그룹
- PPN(Physical Page Number) : 물리적인 페이지 주소
- PBN(Physical Block Number) : 물리적인 블럭 주소
- hot pages : 짧은 시간 안에 자주 업데이트되는 페이지
## Abstract
- 본 논문은 MP3, 디지털 카메라 등의 SSD 기반 어플리케이션의 특성에 기반하여 더 나은 성능과 내구성을 제공하는 새로운 FTL 알고리즘을 제안하고자 함.
## Introduction
- NAND 플래시 메모리는 비휘발성, 낮은 전력 소모, 물리적인 충격에 대한 내성, 신뢰성과 같은 장점에 기반해 다양한 모바일 기기에 탑재되고 있음. 2004년 이후로 지속적으로 저장공간당 비용이 감소함에 따라 특히 노트북에서 HDD의 경쟁자 / 대안으로 자리매김함.
- 그러나, HDD와 다르게 erase-before-write라는 특성을 가져 새 데이터를 쓰기전에는 기존의 데이터를 삭제하는 과정이 필수적임. 이로 인해 FTL 계층이 필수적임. FTL은 읽기 / 쓰기 / 지우기 연산을 수행하고, 운영체제 상의 논리적인 주소를 실제 NAND 플래시 메모리의 주소 상으로 매칭시키는 역할을 수행함.
- FTL의 핵심적인 역할은 NAND 플래시 메모리의 이러한 기술적인 특성을 숨기고(추상화), 최상의 성능과 수명을 가지도록 하는 것임. 이러한 역할은 탑재되는 기기에 따라 제약이 달라짐.
  - 예시 1 : 모바일 디바이스(휴대폰, PDA) 등에 탑재되는 NAND 플래시 메모리는 FAT16/32와 같은 파일 시스템 아래에 block device driver 형태로 구현됨. -> 성능과 함께 운영체제 호환성 또한 중요. 호스트의 성능에 따라 RAM, CPU 성능의 제한은 널럴할 수 있지만 요즘의 멀티미디어 어플리케이션에서는 훨씬 높은 성능을 요구함.

  - ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%202.png)<!-- {"width":239} -->
- 전용 디바이스를 이용하면 읽기 / 쓰기 속도를 극대화할 수 있지만 제한된 컴퓨팅 파워와 메모리를 고려하여 설계해야함. 특히 sequential / random 접근 + 데이터의 길이가 핵심적임.

- NAND 플래시 메모리의 영역이 확장됨에 따라, 다양한 요구사항을 유연하게 충족시킬 수 있는 FTL 아키텍처의 중요성이 커짐.
- 논문에서 제안하고자 하는 아키텍처는 공간적 / 시간적 지역성이라는 두 파라미터를 이용하여 유연한 매핑 알고리즘을 제공함.

## Background
### NAND Flash Architecture
![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%204.png)<!-- {"width":248} -->
- NAND 플래시 메모리는 2KB 공간 + 64B 메타데이터로 구성된 페이지 * 64개로 구성된 블럭을 일정량만큼 가지고 있음.
- 읽기 / 쓰기 연산은 **페이지** 기반으로 수행되지만, 지우기 연산은 **블럭** 단위로 수행됨.
  - **읽기** 연산을 위해서는 명령 코드와 페이지 순번이 I/O 핀을 통해 입력됨. **Page Read**(==🟢20us==)만큼의 레이턴시 뒤, 선택된 페이지가 페이지 및 스페어 레지스터로 로드됨. 최종적으로, 로드된 데이터는 I/O 버스를 통해 시스템 메모리로 로드됨. 스페어 데이터는 배드블럭, ECC와 같은 부수적인 정보를 포함함.
  - **쓰기** 연산을 위해서는 명령 코드와 페이지 순번, 그리고 데이터가 시스템 메모리로부터 페이지 및 스페어 레지스터로 제공됨. **Page Program**(==🟡200us==)만큼의 레이턴시 뒤, 데이터가 지정된 페이지에 작성됨.
  - **지우기** 연산을 위해서는, 명령 코드와 **블럭** 주소가 I/O 핀을 통해 입력됨. **Block Erase**(==🔴1.5ms==)만큼의 레이턴시 뒤, 지정된 블럭이 지워짐.
- HDD, SRAM, DRAM과 다르게, 쓰기 연산의 레이턴시가 읽기 연산의 레이턴시에 비해 김. 그리고 쓰기 연산은 보통 지우기 연산을 동반하기에 실제로는 훨씬 더 느림.
- 추가적으로, 쓰기-지우기 사이클의 수명은 100,000회 정도로 제한되기 때문에, 성능과 수명 모두를 위해 쓰기 및 지우기를 최소화할 필요가 있음.
### FTL Concepts
- 일반적인 FTL 계층은 NAND 플래시 메모리를 메타데이터 블럭과 유저 데이터 블럭으로 분할함. 
- 메타데이터 블럭의 구성
  - 초기 / 런타임 중의 배드 블럭을 대체하기 위한 **Reserved Block** 
  - 논리적 주소를 물리적 주소로 변환하기 위한 **Map Block** 
  - 쓸 데이터를 일시적으로 저장하는 **Write Buffer Block**
- 유저 데이터 블럭은 고정된 수의 데이터 블럭으로 구성됨.
- 4개의 페이지로 구성된 블럭(논리 주소 3, 1, 1, 3)에 대해 **Page Program** 요청이 발생하는 예시
  - 1. **Map Block**을 참조하여 논리적 주소를 물리적 주소로 변환함.
    2. 유저 데이터 블럭이 다른 데이터로 점유되어 있으나, 덮어쓰기 연산은 허용되지 않음. -> **Write Buffer Block**에 수신한 데이터를 임시로 저장함.
    3. 논리 주소 (3, 1)에 대한 두 개의 쓰기 연산은 **Write Buffer Block**에 저장됨.
    4. 논리 주소 (1, 3)에 대한 두 개의 쓰기 연산은 덮어쓰기 연산이 금지된 관계로 3 / 4 페이지에 저장됨.
-> 1 / 2 페이지는 유효하지 않은 데이터, 3 / 4 페이지는 유효한 데이터. 
-> 결과적으로 4번의 **Page Read** 연산 / 4번의 **Page Program** 연산 / 2번의 **Block Erase** 연산이 발생함. 
- **Write Buffer Block**가 유효하지 않은 상태일 경우, 새 페이지를 만들어야 함. 이 경우, **Write Buffer Block**에서 병합을 요청하여 새 페이지를 만듦. 여유 블럭에 **Write Buffer Block**과 기존 페이지를 병합하여 새 페이지를 만듦.
#### -> 병합 연산은 비싸니까 최소화해야 함.
## Mapping Scheme And Related Works
- 논리 주소 -> 물리 주소 매핑 기법 
  - **Page Mapping** : 페이지 기반으로 NAND 플래시 메모리를 관리함. 램과 NAND 플래시 메모리 양쪽에서 매핑 테이블을 관리함. 발견한 페이지에 값이 기록되어 있다면, 유효하지 않은 값으로 간주하고 다른 페이지에 기록함.
    - 예시 : LPN 5에 쓰기 명령이 FTL로 입력되는 경우
    - ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%205.png)<!-- {"width":414} -->
    - 1. 매핑 테이블에서 LPN 5에 대응하는 PPN 2를 찾음. 
      2. PPN 2가 유효한 값으로 채워져 있는 경우, 남는 빈 공간 PPN 5에 데이터를 씀.
      3. 매핑 테이블을 업데이트함.
    - **Pros** 
      - erase-before-write 연산의 최소화 -> 성능 및 수명 향상.
      - 데이터를 유연하게 관리할 수 있음.
    - **Cons**
      - 유효하지 않은 페이지를 늘리기 때문에 **Garbage Collection**이 필수적임.
      - 메모리, NAND 플래시 메모리 양쪽에서 큰 매핑 테이블을 요구함. 
      - 임베디드 시스템과 같이 비용적 제약이 큰 경우 사용하기 힘듦. -> Map Table Caching
  - **Block Mapping** : 블럭 기반으로 NAND 플래시 메모리를 관리함. LPN을 논리적 블록 주소(LBN)과 페이지 오프셋으로 나눠 저장함. LBN을 통해 여유 페이지를 포함하는 블럭을 찾고 페이지 오프셋으로 정확한 페이지 주소를 찾음.
    - 예시 : LPN 5에 쓰기 명령이 FTL로 입력되는 경우
    - ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%206.png)<!-- {"width":340} -->
    - 1. LPN 5가 LBN 1, 페이지 오프셋 1로 변환됨.
      2. LBN 1에 대응하는 PBN 0에 오프셋 1을 더해 PBN 1을 찾음.
      3. PBN 1이 유효한 값으로 채워져 있는 경우, 비어있는 다음 블럭을 찾음.
      4. 동시에, PPN 1이 속한 블럭의 다른 페이지 역시 PPN 1과 같은 블럭에 위치해야 하기 때문에 여유 블럭으로 복사됨.
    - **Pros**
      - 매핑 테이블에 LBN만 기록하기 때문에 크기를 크게 줄일 수 있음.(1MB -> 16KB)
      - 다양한 임베디드 시스템에도 탑재할 수 있음.
      - 순차적 쓰기에 대해 더 나은 성능을 제공함.
    - **Cons**
      - 탑재되는 시스템의 논리적 페이지 오프셋과 물리적 페이지 오프셋이 동일해야 함.
      - 블럭째로 복사하는 과정에서 erase-before-write 연산이 잦아짐 -> 수명 및 성능 저하
      - 무작위 쓰기에 대해 상당한 성능 감소가 수반됨.
  - **Hybrid Mapping** : 매핑 테이블 크기 감소보다는 블럭을 덮어쓰는 과정의 오버헤드를 줄이는 방향으로 타협한 매핑 기법. 대표적으로 로그 기법이 있음. 플래시 메모리의 일부를 덮어쓰기 연산을 위한 로그를 기록하는 블럭으로 사용함. 동일한 페이지에 대한 덮어쓰기 작업이 발생할 경우, 빈 페이지에 기록하고 기존의 데이터는 유효하지 않게됨. 
    - 예시 : PBN 0이 LPN 4, 5, 6, 7을 포함할 때, LPN 5, 7, 7, 5에 쓰기 요청이 발생하는 경우
    - ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%207.png)<!-- {"width":386} -->
    - 1. LPN 7, 5에 대한 요청은 처음 LPN 5, 7에 대한 요청을 덮어씀.
      2. 여유 페이지가 없거나, 요청된 페이지가 포함된 블럭이 이전 논리 블럭에서 변경된 경우, 로그 블럭과 대응하는 블럭이 병합됨.
      3. 병합된 블럭이 유효한 블럭이 됨. 원래 블럭과 로그 블럭은 여유 블럭이 됨.
    - **Pros**
      - 순차적 쓰기 연산, 무작위 접근 모두에서 우수한 성능. 로그 블럭에 연속해서 쓰고 페이지 매핑 정보만 유지함.
      - 적은 수의 로그 블럭만 사용하기 때문에 매핑 오버헤드가 낮음.
    - **Cons**
      - 모든 로그 블럭을 사용한 상태에서, 로그 블럭과 관계없는 쓰기 요청이 발생할 경우, 로그 블럭 중 하나를 대응하는 블럭과 병합해 여유 블럭을 확보해야 함.
      - 로그 블럭이 하나의 데이터 블럭과만 연관이 있어 로그 블럭 활용성이 낮을 수 있음.
      - 사용하지 않은 여유 페이지 수에 따라 병합 연산의 횟수가 늘어남
  - **FAST**(Fully Associative Sector Translation) : 로그 블럭을 모든 데이터 블럭에서 공유함. 모든 쓰기 요청이 로그 블럭에 기록됨. 연속적인 로그 블럭을 이용해 연속 쓰기를 다룸. 
    - **Pros**
      - 병합 연산을 크게 지연 -> 전반적인 성능 및 수명 향상
    - **Cons**
      - 단일 로그 블럭에 여러 데이터 블럭과 연관된 페이지가 포함되어 있어 병합 연산이 더 자주 수행될 수 있음.
      - 병합 작업의 시간이 길어질 수 있음. 
  - **Flexible Management Scheme by Chang and Cuo [2004]** : 대규모 NAND 플래시 메모리를 위한 유연한 기법. 고정된 크기의 매핑 유닛을 활용하는 기존의 방식과 달리, 가변적인 크기의 매핑 유닛을 사용함. 메모리 요구사항을 낮추고, 성능 개선과 수명 향상 등의 효과가 확인됨. 
    - **Pros**
      - 메모리 요구 사항 낮춤 / 성능 개선 / 수명 향상
    - **Cons**
      - 워크로드 패턴에 의존적임.(최악의 경우에도 기존과 유사하게 나옴)
  - **N to N + M Mapping by Kang et al [2006]** : 인접한 N개의 논리 블럭으로 구성되는 슈퍼블럭이 물리적인 페이지 레벨에서 N개의 블럭 + M개의 로그 블럭으로 매핑됨. (로그 블럭의 수는 가변적) 새 블럭이 슈퍼 블럭에 할당될 경우, M이 늘어남. 로그 블럭과 데이터 블럭 간에 병합이 발생할 경우, M이 줄어듦. 슈퍼 블럭 내부의 페이지는 물리적 블럭의 위치를 가리지 않고 매칭됨.
    - **Pros**
      - 다른 방식과 비교해 GC 오버헤드가 40%까지 감소함.
    - **Cons**
      - 매핑 테이블의 크기가 커짐.
      - 파라미터 N, M이 다양한 어플리케이션의 요구사항에 맞춰 조정할 수 없음.
  #### -> 본 논문은 마지막 기법의 파라미터 공간을 효과적으로 탐색할 수 있는 방법에 대해 다룰 것임.
## THE PROPOSED APPROACH: FLEXIBLE GROUP MAPPING
- 기본적인 아이디어 : 공유하는 로그 블럭의 수를 파라미터 N을 통해 조정하고, 자주 업데이트되는 블럭에 대한 로그 블럭의 병합을 몇 번까지 지연시킬지를 파라미터 K를 통해 조정하는 것.
- 최적 파라미터는 대상 애플리케이션의 사용 패턴으로부터 도출됨.
- 데이터 블럭 그룹은 N개의 연속된 블럭으로 구성됨. N은 인접한 블럭 간의 연관성을 의미함.
- N = 1 -> 로그 블럭 기법 / N = 모든 NAND 플래시 메모리 블럭수 -> FAST
- 로그 블럭 그룹 : 특정한 데이터 블럭 그룹과 연관된 로그 블럭의 그룹
- K : 로그 블럭 그룹에 추가될 수 있는 최대 로그 블럭의 수.(로그 블럭을 추가해 병합을 지연하는 것) 동시에 시간적 지역성의 유형을 결정함. hot pages의 경우, 병합 연산 없이 유지하는 것이 유리함. 이 경우, 병합 연산은 최대 K의 값만큼 지연됨. 
- **Switch Merge** 예시 : (1, 1, 14, 15, 2, 2, 3, 3)의 쓰기 명령이 발생함. 로그 블럭 0은 유효한 값과 유효하지 않은 값이 섞여있음.
- ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%208.png)<!-- {"width":414} -->
- 1. 이 경우, 로그 블럭과 데이터 블럭을 병합하는 대신 로그 블럭을 추가하는 것이 유리함.
- 2. 반면에, 이 상태에서 (24, 25, 26, 27)의 쓰기 명령이 발생할 경우, 로그 테이블을 유효한 블럭으로 업데이트하고, 기존의 블럭을 로그 블럭으로 취급하여 병합하는 것이 유효한 페이지를 복사하는 것보다 효율적임.
- 로그 블럭을 추가하는 것은 항상 유리한 것처럼 보이지만, 실제로는 여유 블럭의 수에 의해 제한됨. K를 단순히 늘리기만 하면, 로그 블럭의 할당을 가지고 데이터 블럭 간의 경쟁이 발생하며, 병합 연산을 오히려 늘릴 수도 있음. 
  - **Volunteer Merge Operation** : 병합할 필요가 없더라도, 자발적으로 병합하여 로그 블럭을 놓아주는 것. -> 더 나은 사용자 경험
  - N과 병합 비용은 정비례함.
### Write Scheme
- 논리 주소 -> 물리 주소 변환을 위해 필수적인 매핑 테이블
  - **DBMT**(Data Block Mapping Table) : LBN을 인덱스로 하는 PBN 배열.
  - **LBMT**(Log Block Mapping Table) : 데이터 블럭 그룹 번호와 PBN, PBN은 곧 데이터 블록 그룹의 로그 블럭 번호임.
  - **LPMT**(Log Page Mapping Table) : LPN, PPN과 데이터 블럭 그룹 번호. 로그 블럭의 페이지 매핑 정보 포함함.
- 부수적인 매핑 테이블
  - **BBMT**(Bad Block Mapping Table) : 초기 / 실행 중에 생긴 배드 블럭을 여유 블럭으로 대체하기 위한 테이블.
- 예시 : 쓰기 연산의 실제 과정(N = 4, K = 2 )
- ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%209.png)<!-- {"width":476} -->
- 1. 요청받은 데이터 블럭 그룹 번호와 연관된 로그 블럭이 있는지 확인함.
     1. 연관된 로그 블럭이 없는 경우(LBMT), 여유 블럭 풀로부터 새 로그 블럭을 할당함.
     2. 결과적으로, LPMT에 LPN과 연관된 PPN을 기록함. (LPN에서 LPN + Num of Pages만큼)

     3. 연관된 로그 블럭이 있지만 충분한 여유 페이지가 없는 경우, 여유 블럭 풀로부터 새 로그 블럭을 할당함.(LPN 11, 12 -> PBN 300, LPN 13, 14 -> PBN 400)
     4. 성공할 경우, LBMT에 PBN을 기록함.
     5. LPMT에 LPN, PPN을 기록함.

## Performance Model And Analysis
- FGM은 그룹의 사이즈를 결정하는 파라미터 N을 통해 공간적 지역성만 고려할 뿐만 아니라, 병합을 지연시키는 파라미터 K를 통해 시간적 지역성 또한 고려함. 이 파라미터들은 FTL을 유연하게 설정할뿐만 아니라 특정한 NAND 플래시 메모리 어플리케이션에 따라 조정될 수 있음. 그러나 별도의 파라미터 공간 탐색 알고리즘 없이는 최적 파라미터를 찾기 힘듦. 이 논문에서는 주어진 제약 안에서 효과적으로 파라미터 셋을 탐색할 수 있는 방법과 성능 모델을 소개할 예정임.
### Workload Analysis
- 저장소 접근 패턴은 요청 밀도 관점에서 조사할 수 있음.
- 쓰기 요청 시퀀스 -> $R$ $=$ ⟨$R_0$,  $R_1$,. . . , $R_{M-1}$⟩ 
- 쓰기 요청 윈도우 -> $W$ $=$ ⟨$R_{j ·|W|}$,. . . , $R_{(j +1)·|W|−1}$⟩
- 요청 밀도 -> $RD_{i,j}$ $=$ $\frac{C_{i,j}}{|W|}$
- $\sum\limits_{i\in\text{all LBNs}}RD_{i,j}=\frac{1}{|W|}\sum\limits_{i\in\text{all LBNs}}C_{i,j}=1 \ for \ any \ requests \ window \ W_{j}$

- 예시 : $R_{0}$  ~ $R_{19}$ 까지 쓰기 요청 시퀀스가 발생할 경우($|W|=4$)
- ![](%E1%84%80%E1%85%A9%E1%84%80%E1%85%B3%E1%86%B8%E1%84%8F%E1%85%A5%E1%86%B7%E1%84%91%E1%85%B2%E1%84%90%E1%85%A5%E1%84%80%E1%85%AE%E1%84%8C%E1%85%A9%20%E1%84%87%E1%85%A1%E1%86%AF%E1%84%91%E1%85%AD%20%E1%84%82%E1%85%A9%E1%86%AB%E1%84%86%E1%85%AE%E1%86%AB%20%E1%84%8C%E1%85%A5%E1%86%BC%E1%84%85%E1%85%B5/image%2013.png)<!-- {"width":317} -->
- 1. LBN 0 ~ 3까지 영향을 받음. 
  2. 각 $RD_{ij}$ 의 값은 N의 최적값을 결정하는 단서로 활용할 수 있음. $RD_{ij}$ 가 높을 경우, N을 줄여도 되고, $RD_{ij}$ 가 낮을 경우 N을 키워야 함. N의 값은 $RD_{ij}$ 의 역수에 비례함.
  3. N이 클수록 여러 블럭이 하나의 로그 블럭을 공유하기 때문에 효율이 높아짐. 그러나 동시에 병합 비용 또한 상승. 
  4. 히스토그램 활용해서 찾기 -> 실제로 효율적임.
  5. K는 요청창 $W_j$ 에서 각 LBN의 시간적 지역성을 측정해서 찾음. 시간적 지역성 -> LBN i의 1개 이상의 페이지가 1번 이상 업데이트된 경우, 직전 윈도에 쓰인 요청창에 쓰인 페이지 1개 이상이 현재 윈도우에서도 업데이트된 경우
  6. $K_{i,j}$ 는 $W_0$ ~ $W_J$ 까지 LBN i에 대해 조건을 충족시키는 요청의 횟수.

### Performance Analysis
- 편의 상, 모든 쓰기 요청의 사이즈는 1개의 섹터만으로 한정함. 1개 이상의 섹터에 대한 쓰기는 1개 섹터에 대한 쓰기의 리스트로 변환할 수 있음.
- $A_k =$ 쓰기 요청 $R_k$ 가 접근한 블럭.
- $AG_k =$ 쓰기 요청 $R_k$ 가 접근한 DGN. 항상 $A_k$ 포함함.
- $SA(W_j), SAG(W_j), L(AG_k) =$ 요청창 $W_j$ 에 의해 활성화된 블럭 / 블럭 그룹 집합. / $AG_k$ 와 연관된 로그 블럭의 수.
- $\sum\limits_{AG_k\ \in\ SAG(W_j)} L(AG_k) > LB$ -> $|SAG(W_j)| * K^\delta > LB$ 

  - (활성 블럭 그룹 집합에 속한 모든 블럭의 로그 블럭의 합이 시스템의 최대 로그블럭 수를 추월한 경우,), 병합 연산이 발생함. 이 방정식은 $|SAG(W_j)| * K^\delta > LB \ for \ \delta (0<= \delta < 1)|$ 로 치환 가능.  ($\delta$ = 입력 패턴의 업데이트 빈도)
  - 쓰기 요청 시퀀스 $W_j$ 의 상당수가 $AG_k$ 에 접근할 경우, $AG_k$ 는 더 많은 로그 블럭을 소유함. 이 경우, $\delta$ 의 값은 0으로 수렴하여 조건의 엄격함을 낮춤.
  - 쓰기 요청 시퀀스 $W_j$ 가 여러 활성 그룹에 산발적으로 접근할 경우, $AG_k$ 는 더 적은 로그 블럭 소유함. 이 경우, $\delta$ 의 값은 1으로 수렴하여 조건의 엄격하게 함.
- $|SAG(W_j)| =$ 요청창 $W_j$ 의 활성화된 블럭 그룹의 집합에 속한 블럭의 수 -> $|SA(W_j)/N^\epsilon \ for \ 0 < \epsilon <= 1$  로 표현할 수 있음. $\epsilon$ 은 입력패턴의 연관성을 의미하며, 연관성이 높을 경우, 활성 블럭은 연속된 데이터 블럭일 수도 있음.
- 입력 패턴이 충분히 크고, 변동이 적을 경우 $SA(W_j)$ 는 상수로 취급할 수도 있음. 그러므로, $|SAG(W_j)|$ 는 $|SA(W_j)/N$ 과 유사하게 수렴함. 결과적으로 $\epsilon$ 의 값은 1로 수렴함.
- 입력 패턴의 연관성이 낮을 경우,$\epsilon$ 의 값은 0으로 수렴함.
- $C * K^\delta / N^\epsilon > LB \ for \ a \ constant \ C$ <- 위 실험 결과로부터 도출되는 값. 이 값을 통해 NAND 플래시 메모리 응용 프로그램의 성능을 분석할 수 있음.
  - MP3의 경우, 요청창의 크기가 작아 $\delta$ 의 크기가 0에 수렴함. 또한, 연관성 또한 낮아 N 역시 0에 수렴. 이 경우, N, K가 중요한 파라미터가 아님.
  - PC 응용 프로그램의 경우, MP3보다 요청창의 크기가 크고 $\delta$ 의 크기가 1로 수렴함. 연관성 역시 높으나, $\epsilon$ 의 크기가 낮음. 이 경우, K가 클 경우 많은 병합 연산을 유발해 성능을 저하시킴. N은 중요한 파라미터가 아님.

### Memory Requirement Analysis
- 활성 로그 블럭 그룹은 세 가지 자료구조를 가짐. 
  - 64N개의 페이지에 대한 매핑 테이블. 크기는 N의 값에 비례함. -> c1
  - 관련된 1개 이상의 로그 블럭에 대한 정보. 로그 안의 유효한 페이지의 수에 근접하는 적은 상수 값을 가짐. 크기는 K의 값에 비례함. -> c2
  - 로그의 수를 근사할 수 있게 하는 수. -> c0
  - $Memory \ requirement \ = \ |SAG(W_j)| * c0 * c1N * c2K$
